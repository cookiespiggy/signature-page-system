一 业务关键理解：从工具到法律生产基础设施

1.1 行业语境：法律文书生产的最后一公里

在 IPO、并购、基金设立等跨境/境内交易中，签字页是整套法律文件的"终点"——所有谈判、条款、合规审查的成果最终凝结为一个个签字页上的变量值。一个典型交易涉及 3~10 个签字页模板，涵盖自然人股东、机构股东、律师事务所等多种签署主体，每个模板 6~20 个变量，其中 30%~50% 为跨模板共享的核心字段。

然而，这个"最后一公里"恰恰是整个法律文档数字化链条中最薄弱的环节。律所的合同管理系统解决了合同起草，尽调平台解决了尽职调查，但签字页——这个最终交付物——仍然停留在手工填写 Word 文档的阶段。

战略洞察：签字页不只是一个 Word 文件，它是法律服务生产流程的数字化闭环节点。谁掌握了签字页的生成入口，谁就掌握了从"合同起草 → 尽调审查 → 签字交付"的全链路。MVP 解决的是"降本"，但它的战略价值远不止于此。

1.2 核心痛点：三层递进分析

表层痛点：重复劳动

律师需手工填写多份模板中相同的字段。一个 IPO 项目 5~10 个模板 × 8~20 个变量 = 40~200 次手工输入，其中 30%+ 是重复内容。重复录入不仅耗时，且极易产生不一致。

深层痛点：领域语言碎片化

"目标公司"vs"被投公司"vs"标的企业"vs"发行人名称"——同一个法律概念在不同律所、不同模板中有完全不同的表达。缺乏统一的领域本体（Ontology），使得跨模板、跨项目的变量管理成为噩梦。

根层痛点：生产流程不可控

没有标准化的签字页生产流程：模板版本混乱、变量填写无校验、生成过程不可中断不可追溯。质量完全依赖个人经验，无法规模化复制。

1.4 AI 赋能：三个精确场景

AI 在本系统中的定位非常明确：不产生数据，只产生建议。所有 AI 输出必须经过「规则层 → 交叉验证层 → 人类确认层」三重关卡后才能落地写入数据库。

当 LLM 服务不可用（超时、网络故障、重试耗尽）时，系统不会崩溃，而是优雅降级到规则层。以 AI-2 去重为例：

3.6 审计闭环：AILog 全链路记录

每次 AI 调用写入 ai_logs 表，记录 prompt/response/duration_ms/validation_result/user_action，支持事后追溯与质量分析。这为后续的 Eval 和 Tracing 基建预留了零成本迁移接口。

四 结构化输出工程化：解决 LLM 返回 JSON 不可靠

4.1 核心挑战

LLM 本质是文本生成器，其返回的 JSON 经常出现：格式污染（```json``` 包裹）、语法错误（尾部逗号）、Schema 偏离（字段名拼错、类型不匹配）、幻觉数据（编造不存在的 key）。这些不是靠"写好 prompt"能根治的，必须在工程层面系统性解决。

4.2 双路径自适应策略

路径 A：原生结构化输出

适用于支持 format/json_schema 的 Provider（如 OpenCode + mimo-v2.5-free）。LLM 采样阶段即被 Schema 约束，输出几乎 100% 合规。

raw = await provider.send_message(
    prompt,
    format={"type": "json_schema",
            "schema": schema.model_json_schema()})
data = json.loads(raw)
return schema.model_validate(data)

路径 B：Prompt 嵌入 + 兜底提取

适用于不支持原生结构化的 Provider。Schema 嵌入 system prompt，JSON 提取器从响应中恢复结构化数据。

raw = await provider.send_message(
    prompt, system=_build_structured_system(...))
json_str = _extract_json(raw)
return schema.model_validate(
    json.loads(json_str))

4.3 完整容错链：五层防御

第 1 层：LLM 调用 — 超时 30s → 自动重试（最多 3 次）→ 网络异常重试 第 2 层：JSON 解析 — _extract_json() 处理格式污染 → json.loads() 语法校验 → 失败追加格式修正提示 第 3 层：Pydantic 校验 — model_validate() 严格类型检查 → ValidationError → 重试时动态追加错误信息+Schema 第 4 层：Guardrail 交叉验证 — trust_level 评级 → 冲突检测 → 异常 AI 结果标记 LOW 第 5 层：降级兜底 — 3 次重试全失败 → AIServiceUnavailableError → 规则层结果仍可用

5.2 关键设计模式

快照隔离 (Snapshot Isolation)

ProjectTemplate.variables_snapshot_json 在选择模板时复制 Template.variables_json。后续模板升级不影响已有项目的变量结构和生成结果。前端通过 template_version 对比检测更新。

乐观锁 (Optimistic Locking)

Variable.updated_at 作为隐式版本号。批量保存时逐行比对：成功行写入、冲突行返回错误报告（含当前值），不整体回滚。前端局部刷新冲突行后重试。

源追踪 (Source Tracking)

Variable.source_template_ids 记录变量来源模板 ID 列表。移除模板时自动清理孤立变量（source_template_ids 为空则删除），保留共享变量。CASCADE + SET NULL + 清理守卫三重保障。

多值变量展开 (Multi-value Expansion)

is_multiple=True 的变量在 Variable 表中展开为多行（key_1, key_2, ...），生成时由 _build_render_context() 按 key 前缀聚合回列表传入 docxtpl 循环渲染。

设计哲学：所有历史数据均可追溯（快照 + 版本号 + 审计日志），所有变更都可控（乐观锁 + 部分成功），所有数据关联都级联安全（CASCADE + SET NULL + 清理守卫）。

6.3 核心流程：变量乐观锁保存

「逐行校验、部分成功」—— 不回滚整个事务，成功行写入，失败行返回详细错误：

def save_variables(db, project_id, items):
    success, errors = [], []
    for item in items:
        var = _find_variable(db, project_id, item["key"])
        client_dt = _normalize_dt(item["updated_at"])  # 乐观锁比对
        db_dt     = _normalize_dt(var.updated_at)
        if client_dt != db_dt:
            errors.append({"row": i, "message": "数据已被其他操作修改"})
            continue  # 跳过冲突行，不回滚
        fmt_error = validate_value(key, value, var.data_type)
        if fmt_error:
            errors.append({"row": i, "message": fmt_error})
            continue
        var.value = value  # 成功行写入
    db.commit()
    return {"success": success, "errors": errors, "summary": {...}}

七 实现取舍与关键决策

决策点选择舍弃理由与权衡
数据库SQLite + WALPostgreSQL / MySQLMVP 零部署成本，单文件便于开发测试迁移。WAL 支持并发读。代价是写入并发受限（max_workers=2），生产环境需迁移 PG（仅改 database.py）。
异步生成ThreadPoolExecutor + EventCelery / BackgroundTasks① Celery 引入 Redis/Broker 依赖，MVP 过重；② BackgroundTasks 无法精确取消；③ 线程池 + Event 毫秒级取消，独立 DB Session 避免生命周期冲突。
前端状态React useStateRedux / ZustandMVP 状态复杂度不高（3 个 Step + 列表），引入外部状态库属于过度工程。V2 团队协作再引入。
LLM 抽象LLMProvider 工厂模式硬编码 LLM 调用统一 send_message() 接口，运行时按环境变量切换。Mock Provider 支持离线开发和 CI。
结构化输出双路径自适应单一 Function Calling原生 json_schema 的模型直接传 Schema；不支持的走 Prompt 嵌入 + JSON 提取。重试时动态追加格式修正。
多值变量多行存储 (key_1, key_2)JSON 数组避免 JSON 在 Excel 导入导出、逐行操作时的复杂性。生成时动态聚合为列表。
变量去重规则 + AI + 交叉验证纯 AI 去重规则层保证零误判；AI 层覆盖规则盲区；交叉验证给每条建议标 trust_level，人类做最终决策。
预置模板is_preset + 403 拒绝软删除 / 404防止误删核心业务模板。403 比 404 更明确传达「不允许」语义。
最关键的架构决策：选择 ThreadPoolExecutor 而非 FastAPI BackgroundTasks，原因有三：① docxtpl 是同步 CPU/IO 密集型库，不适合 asyncio event loop；② threading.Event 提供毫秒级取消响应；③ 每个 worker 线程创建独立 SQLAlchemy Session，避免与请求级 Session 产生 SQLite 写锁竞争（max_workers=2 控制并发度）。

十 AI 工程演进方向研判

在确定 AI 侧深度投入方向前，需回答三个常被提出的问题：何时引入 Agent？是否建设 LLM judge / eval / tracing / 可视化？是否投入 ontology（本体）与输出稳定性？以下基于业务场景本质给出研判结论。

10.1 研判标尺

衡量标准（唯一）：一个 AI 方向是否值得投入，取决于它能否更直接地 —— ① 降低重复劳动；② 降低错误率；③ 增强可追溯。与"AI 不产生数据，只产生建议"的安全原则同向者优先。本系统的本质是确定性的法律文书流水线，在 to-B 律所场景中，可靠、可解释、可审计、合规是首要价值。

10.2 三个方向逐项研判

方向一：Agent 化 暂不上

当前三个场景均为 one-shot 结构化输出，流程确定，不存在需要自主规划、多步探索的空间。Agent 的价值在于"任务路径不确定、需自行决定下一步"，与签字页生成的确定性恰好相反。勉强可落地的 agentic 场景仅两个半成品：AI 自动填充（用 RAG/规则检索即可）、生成后质量审查（本质仍是单步比对）。触发条件：当业务扩展到"自动起草整份法律文件 / 跨文档一致性推理"时再评估。

方向二：LLM judge / eval / tracing / 可视化 部分采纳

子方向结论理由
tracing / 审计可视化轻量做已具备 AILog 全链路审计表，基于 AILog 做轻量看板（调用次数/成功率/耗时/采纳率），定位为"运维可观测"，够用即可。
LLM judge / 正规 eval暂缓3 个辅助型 AI 场景，独立 eval 投入产出比偏低。应先做"反馈闭环"（采纳/拒绝 → 调阈值，穷人版 eval），待数据积累充分后再上重型 eval。

方向三：Ontology + 输出稳定 AI 侧第一重点

输出稳定/去幻觉已有扎实基础（双路径结构化输出、Pydantic 校验、规则交叉验证、trust_level），守住现状即可。Ontology（法律变量本体）是性价比最高的方向：直接命中核心痛点（同一法律概念多种字段名），且已有雏形（VARIABLE_REGISTRY + CustomVariable.aliases + 别名匹配），属"升级"而非"从零造轮子"。

核心论点：与其让 LLM"更聪明地猜"，不如把领域知识沉淀进 ontology 让它"少猜"。本体越完善，规则层（高可信、零误判）覆盖面越大，需要 LLM 兜底的盲区越小 —— 用确定性手段提升准确率，可解释、可审计、天然合规。

10.3 优先级结论

方向定位核心理由
业务功能（PG/认证/PDF/套表组）第一优先（产品工程）直接交付价值
① Ontology 升级AI 侧第一重点确定性解决核心痛点，降低 LLM 依赖
② 采纳率反馈闭环 + 轻量审计看板AI 侧第二重点合规留痕 + 为 eval 积累数据
③ 输出稳定/去幻觉守住现状基础已扎实，边际收益递减
④ LLM judge / eval暂缓先以反馈闭环替代
⑤ Agent 化暂不上业务确定性强，易引入不可预测性

10.4 Ontology 升级落地路径

第一步：本体 Schema 统一

每个变量定义为结构化条目：标准 key / 中文 label / category / data_type / 别名集 / 校验规则 / 适用主体（自然人股东/机构股东/律所）/ 可信来源。别名集中管理，扩大去重"规则层"覆盖面。

第二步：反馈回写 + 规则下沉

用户每次采纳/拒绝 AI 去重建议时，将确认过的"key ↔ 别名"关系回写本体，使规则层自我成长；AI-3 校验规则统一下沉至本体驱动。

十三 开发方法论总结

13.1 AI Agent + Session 状态机

本次 MVP 采用独特的开发模式：将复杂系统拆分为 7 个 Session，每个 Session 有明确的 DoD（完成标准）、产出文件和依赖关系，通过 Handoff 文件实现跨 Session 的知识传递。

AGENTS.md（项目唯一入口） ├── Session 1a: 后端脚手架 + 数据模型 + 变量注册表 │ └─> Session 1b: LLM 抽象层 + 项目 API │ ├─> Session 2: 模板管理 + AI 解析 │ ├─> Session 3: 变量填写 + AI 去重校验 │ └─> Session 4: 异步生成 + 下载 │ └─> Session 5a: 前端列表页 │ └─> Session 5b: 前端详情页 + AI │ └─> Session 6: 联调测试 │ └─> Session 7: 报告 └── .handoff/latest.md（Session 间知识传递）

13.2 关键实践

实践具体做法价值
单一入口文档AGENTS.md 定义 Session 边界、DoD、技术决策新 Session 无需重复沟通上下文
Handoff 机制每个 Session 结束后持久化状态到 .handoff/跨 Session 知识不丢失
方案先行先写 docs/specs/ 实现方案，再编码减少返工，决策可追溯
知识沉淀踩坑经验写入 docs/knowledge-base/AI 可信机制、结构化输出、环境配置可复用
渐进式 AI 集成Mock Provider 先行 → 规则层兜底 → AI 层叠加AI 不可用时系统仍可用
这一方法论使得 7 个 Session 的 ~9,800 行全栈代码 + ~680 行测试 + 完整文档体系 得以在高效协作中高质量交付。每个 Session 都有独立可验证的 DoD，任何一环失败都可精确定位和重做。
