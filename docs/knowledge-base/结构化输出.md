# 结构化输出

> LLM 返回结构化 JSON 的策略与最佳实践。

## 背景

签字页管理系统需要 LLM 返回结构化 JSON（模板解析、变量去重、数据校验）。
不同模型对结构化输出的支持程度不同，需要适配。

## 策略对比

| 策略 | 适用模型 | 当前状态 |
|------|---------|---------|
| **OpenCode 原生 structured output** | `mimo-v2.5-free`（已验证） | ✅ 已启用 |
| **Prompt 嵌入 Schema** | DeepSeek, Mock, OpenAI 等（回退方案） | ✅ 已实现 |

## 策略一：OpenCode 原生 structured output（推荐，mimo-v2.5-free）

`mimo-v2.5-free` 模型**原生支持** `format/json_schema` 参数。
响应中的 `info.structured` 包含验证后的 JSON 数据，无需二次提取。

### 调用方式

```python
provider.send_message(
    prompt="Extract company name and founding year...",
    format={"type": "json_schema", "schema": {...}}
)
# 返回: '{"company": "OpenAI", "founded": 2015}'
```

通过 `opencode_provider.py` 的 `send_message(format=...)` 实现：
- 自动切换到 `mimo-v2.5-free` 模型
- 优先读取 `info.structured` 字段
- 降级：若结构化为空，回退到 text parts 提取

### 已验证结果

- `info.structured` 直接返回符合 Schema 的 JSON
- `finish: "tool-calls"` 表示模型使用 tool calling 成功
- 仅 37 个 output tokens，质量高

### 不支持的模型

| 模型 | 原因 |
|------|------|
| `big-pickle` (DeepSeek) | "Thinking mode does not support this tool_choice" |
| `deepseek-v4-flash-free` | 同上 |
| `mimo-v2-omni-free` | HTTP 500 |
| `nemotron-3-super-free` | HTTP 500 |
| `minimax-m3-free` | HTTP 500 |
| `minimax-m2.5-free` | HTTP 500 |

## 策略二：Prompt 嵌入 Schema（回退方案，DeepSeek/Mock/OpenAI）

在 `call_llm_structured()` 中：

1. 用 `schema.model_json_schema()` 获取 Pydantic 模型的 JSON Schema
2. 通过 `_build_structured_system()` 嵌入到 system prompt
3. LLM 按 Schema 要求输出 JSON
4. 失败时 retry 3 次，每次在 prompt 追加错误信息

适用任何模型，用于不支持原生结构化输出的场景。

## 实现位置

- `backend/app/services/ai_service.py` — `call_llm_structured()` + `_build_structured_system()`
- `backend/app/schemas.py` — 各场景的 Pydantic Schema（`TemplateParseResult` / `VariableDedupResult` / `DataValidateResult`）
