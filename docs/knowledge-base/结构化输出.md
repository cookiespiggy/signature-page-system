# 结构化输出

> LLM 返回结构化 JSON 的策略与最佳实践。

## 背景

签字页管理系统需要 LLM 返回结构化 JSON（模板解析、变量去重、数据校验）。
不同模型对结构化输出的支持程度不同，需要适配。

## 当前默认策略（已实测）

| 优先级 | Provider | Model | 状态 |
|--------|---------|-------|------|
| 1（主） | `github-copilot` | `gpt-5-mini` | ✅ 验证通过，速度快（~14s） |
| 2（备） | `opencode` | `mimo-v2.5-free` | ⚠️ 免费限额有限，耗尽后超时 |

> 主模型失败时自动 fallback 到备用模型，由 `opencode_provider.py` 的多候选逻辑实现。

---

## 策略一：原生结构化输出（推荐）

通过 `format/json_schema` 参数让模型原生返回 JSON，不依赖 prompt 工程。

### 已验证支持的模型

| Provider | Model | 说明 |
|---------|-------|------|
| `github-copilot` | `gpt-5-mini` | ✅ 推荐首选，响应约 14s，`info.structured` 有值 |
| `opencode` | `mimo-v2.5-free` | ✅ 验证通过，但免费次数有上限，耗尽后无限超时 |

### 调用方式

```python
provider.send_message(
    prompt="解析这段模板中的变量...",
    format={"type": "json_schema", "schema": schema.model_json_schema()}
)
# 返回: '{"variables": [...]}'（从 info.structured 提取）
```

### 响应特征

```json
{
  "info": {
    "finish": "tool-calls",
    "structured": { ...符合 schema 的数据... }
  },
  "parts": [...]
}
```

- `finish: "tool-calls"` 表示模型通过 tool calling 完成了结构化输出
- `info.structured` 直接是验证后的对象，无需二次解析

### 验证方法（curl 直接测）

```bash
SID=$(orb -m oh-agent curl -s -X POST http://localhost:4096/session | python3 -c "import sys,json; print(json.load(sys.stdin)['id'])")
orb -m oh-agent curl -s -m 60 -X POST "http://localhost:4096/session/$SID/message" \
  -H 'Content-Type: application/json' \
  -d '{
    "parts":[{"type":"text","text":"返回一个包含 ok 字段的 JSON"}],
    "model":{"providerID":"github-copilot","modelID":"gpt-5-mini"},
    "format":{"type":"json_schema","schema":{"type":"object","properties":{"ok":{"type":"boolean"}},"required":["ok"]}}
  }' | python3 -c "import sys,json; d=json.load(sys.stdin); print('structured=', d['info'].get('structured'))"
# → structured= {'ok': True}
```

## 策略二：Prompt 嵌入 Schema（回退方案）

当模型不支持原生结构化输出时，将 Schema 嵌入 system prompt，让模型按格式返回 JSON。

### 流程

1. 用 `schema.model_json_schema()` 获取 JSON Schema 字符串
2. 通过 `_build_structured_system()` 追加到 system prompt
3. LLM 按 Schema 要求输出 JSON 文本
4. 解析失败则 retry 3 次，每次追加错误信息到 prompt

### 适用场景

- MockProvider（测试）
- OpenAI Provider（直连 API Key 模式）
- 不支持 tool calling 的模型

---

## 不支持结构化输出的模型（已验证失败）

| Provider | Model | 失败原因 |
|---------|-------|---------|
| `opencode` | `big-pickle`（DeepSeek） | `Thinking mode does not support this tool_choice` |
| `opencode` | `deepseek-v4-flash-free` | 同上 |
| `opencode` | `mimo-v2-omni-free` | HTTP 500 |
| `opencode` | `nemotron-3-super-free` | HTTP 500 |
| `opencode` | `minimax-m3-free` | HTTP 500 |
| `opencode` | `minimax-m2.5-free` | HTTP 500 |

> DeepSeek 系列的思考模式（Thinking Mode）不支持 `tool_choice`，因此原生结构化输出不可用。
> 这类模型只能走策略二（Prompt 嵌入 Schema）。

## 多模型 Fallback 机制

`opencode_provider.py` 中实现了候选模型列表 + 自动切换：

```python
# 候选列表（按优先级排列）
candidates = [
    {"providerID": "github-copilot", "modelID": "gpt-5-mini"},   # 主
    {"providerID": "opencode",       "modelID": "mimo-v2.5-free"}, # 备
]
# 主模型失败（任何异常）→ 自动切到下一个候选
```

### 通过环境变量覆盖

```bash
# 指定主模型
OPENCODE_STRUCTURED_PROVIDER=github-copilot
OPENCODE_STRUCTURED_MODEL=gpt-5-mini

# 指定备用模型
OPENCODE_STRUCTURED_FALLBACK_PROVIDER=opencode
OPENCODE_STRUCTURED_FALLBACK_MODEL=mimo-v2.5-free
```

## 免费模型限额问题

`*-free` 后缀的模型（`mimo-v2.5-free` 等）有每日/每月免费请求次数上限。

**限额耗尽的表现**：请求不报错，但无限期挂起，直到客户端超时。

**快速排查**：
```bash
# 用 -m 30 限制 curl 最多等 30 秒
SID=$(orb -m oh-agent curl -s -X POST http://localhost:4096/session | python3 -c "import sys,json; print(json.load(sys.stdin)['id'])")
orb -m oh-agent curl -s -m 30 -X POST "http://localhost:4096/session/$SID/message" \
  -H 'Content-Type: application/json' \
  -d '{"parts":[{"type":"text","text":"hi"}],"model":{"providerID":"opencode","modelID":"mimo-v2.5-free"}}'
# 若 30s 内无响应 → 限额耗尽，换模型
```
