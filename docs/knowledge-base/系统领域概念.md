# 签字页管理系统 — 领域概念与动作

> 本文档解释系统自身的业务概念和操作逻辑，方便理解系统行为背后的设计意图。

---

## 一、核心数据实体

| 概念 | 表 | 一句话解释 |
|------|-----|-----------|
| 项目 (Project) | `projects` | 一次签字页生成任务的聚合根，一个项目包含 N 个模板 + N 个变量 + 1 次生成 |
| 模板 (Template) | `templates` | 签字页的 Word 文件 + 它的变量定义集合，分为预置和自定义 |
| 项目-模板关联 (ProjectTemplate) | `project_templates` | 记录"某项目选了哪些模板"，包含选模板时的变量快照 |
| 变量 (Variable) | `variables` | 项目级别的变量值，按 key 去重（相同 key 合并成一行） |
| 自定义变量注册 (CustomVariable) | `custom_variables` | 用户确认的新变量登记到注册表，**全局生效** |
| 生成任务 (GenerationTask) | `generation_tasks` | 一次文档生成的执行记录，含进度和状态 |
| 生成文件 (GeneratedFile) | `generated_files` | 生成出的 .docx 文件记录 |
| AI 日志 (AILog) | `ai_logs` | 每次 AI 调用的输入/输出/耗时记录，用于审计和调试 |

## 二、模板体系

### 预置模板 vs 自定义模板

| 维度 | 预置模板 | 自定义模板 |
|------|---------|-----------|
| 来源 | 系统自带（3 个 .docx） | 用户上传 |
| 可删除 | ❌ 不可删除（返回 403） | ✅ 可删除 |
| 变量定义 | 写死在 `TEMPLATE_VARIABLE_MAP` | AI 解析生成，用户确认 |
| 更新方式 | 代码部署时自动更新 | 手动上传替换，version++ |

### 模板刷新（Refresh）与 Diff 算法

当模板的 `variables_json` 更新后，已引用该模板的项目需要"刷新"：

1. 对比旧快照和新快照的 key 集合：
   - **新增的 key** → 在项目 Variable 表中插入新行
   - **删除的 key** → 清理 Variable 的 `source_template_ids`，为空则删除整行
   - **保留的 key** → 更新 `required` 状态
2. 刷新后 ProjectTemplate 的 `template_version` 和 `variables_snapshot_json` 更新为新版本

业务含义：律所更新了模板（比如新增一个签字人字段），历史项目可以一键同步，无需手动补漏。

### 变量快照（Snapshot）

模板的变量定义可能随着版本更新而变化。`ProjectTemplate.variables_snapshot_json` 记录了**选模板那一刻**的变量定义，是一个"照片"。

作用：
- 刷新的 diff 依据（旧快照 vs 新快照）
- 即使模板后来变了，也能知道当初选的是什么版本

## 三、变量体系

### 注册表（Variable Registry）

系统的"变量词典"，分为两层：

| 层 | 名称 | 特点 |
|----|------|------|
| 静态注册表 | `VARIABLE_REGISTRY` | 代码写死、git 版本控制、不可变。包含 14 个标准变量 |
| 运行时注册表 | `CustomVariable` 表 + 缓存 | 用户确认新变量时写入，应用启动时加载到内存 |

两者合并后通过 `get_merged_registry()` 查询。

### 已注册 vs 未注册

| 状态 | 含义 | 系统行为 |
|------|------|---------|
| **已注册** (is_registered=true) | key 命中了注册表 | label/data_type/category 以注册表为准，覆盖 AI 猜测 |
| **未注册** (is_registered=false) | key 不在注册表中 | 走交叉验证：尝试别名匹配，不匹配则标 LOW 可信 |

业务含义："已注册" ＝ 系统认识这个字段，它的定义是可靠的、经过确认的。

### 变量去重（Dedup）

**选模板时的自动去重**：当项目新增一个模板时，系统按 key（精确匹配）比对已有变量：
- key 相同 → 该变量的 `source_template_ids` 追加当前模板 ID
- key 不同 → 新增变量行

**AI 去重**：用户触发的语义级去重（`POST /ai-dedup`），LLM 分析变量列表，建议合并语义相同但 key 不同的变量。

### 多值变量（is_multiple）

经办律师可以有多个，系统不存 JSON 数组，而是存为**多行**：
```
handling_lawyer_1 = "张三"
handling_lawyer_2 = "李四"
```

前端默认展示 2 行，用户可动态添加/删除行。生成时系统自动组装为列表传给 docxtpl。

### 乐观锁（Optimistic Lock）

**问题**：多人同时编辑变量时可能互相覆盖。

**方案**：每行 Variable 携带 `updated_at`。用户提交时：

```
PUT /variables
请求体: [{key: "law_firm_name", value: "...", updated_at: "2024-01-01T00:00:00"}]
后端: if db.updated_at != 请求的 updated_at → 409 冲突
      else → 写入 + 更新 updated_at
```

不整体回滚——冲突的行报错，不冲突的行正常写入。

## 四、AI 可信体系

### 四层防御架构

```
LLM 输出 → 程序校验 → 交叉验证 → 人工审核
  第1层       第2层       第3层       第4层
```

每层拦截不同维度的风险，逐层收窄。

---

### 第1层：LLM 输出约束（结构化输出）

**目标**：从源头约束 LLM 的回复格式，防止乱输出。

**实现**：
- 每个 AI 场景绑定一个 **Pydantic Schema**（如 `TemplateParseResult`），发送时附 JSON Schema
- Provider 支持原生结构化输出（如 mimo-v2.5-free）→ 直接传 `format={"type": "json_schema"}`
- Provider 不支持 → 在 system prompt 中嵌入 Schema + 3 次重试 + 格式修正反馈
- 重试耗尽 → 抛出 `AIServiceUnavailableError`，触发降级

**相关代码**：`ai_service.py:call_llm_structured()`

---

### 第2层：程序校验（确定性规则）

**目标**：用硬编码规则和注册表，零成本地校验 LLM 输出。

#### 2a. 注册表匹配

```python
VARIABLE_REGISTRY = {
    "target_company_name": {
        "label": "目标公司名称",
        "category": "company",
        "aliases": ["目标公司名称", "发行人名称", "公司名称", "股份有限公司"],
    },
    # ...
}
```

AI 返回的变量 key 如果在注册表中：
- label / category / data_type / required **以注册表为准**，覆盖 LLM 的猜测
- 注册表中定义的 `aliases` 用于模糊匹配（不区分大小写 + 短词保护）

**关键规则**：包含匹配要求双方字符串长度 **>= 4**，防止"有限"误匹配"股份有限公司"。

#### 2b. 正则校验

```python
VALIDATION_RULES = {
    "natural_shareholder_id_number": r"^\d{17}[\dXx]$",
    "target_company_name": r".+股份有限公司|.+有限责任公司",
    "signing_date": r"^\d{4}年\d{1,2}月\d{1,2}日$",
}
```

这些规则在变量保存时自动执行（`variable_service.py:validate_value`）：
- 命中规则但值不匹配 → 返回格式错误
- 未命中规则 → 跳过校验
- 空值 → 跳过校验

#### 2c. 别名退火（Alias Matching）

注册表的 aliases 设计：
```
"target_company_name" → aliases: ["目标公司名称", "发行人名称", "公司名称", "股份有限公司"]
```

- 精确匹配（不区分大小写）
- **包含匹配**：要求双方 >= 4 字符，如"公司名称"包含"公司"（< 4，跳过），"股份有限公司"包含"股份"（各 2，跳过）
- 匹配成功仅设为 MEDIUM，不自动采纳

---

### 第3层：交叉验证（Guardrail）

**目标**：将 LLM 建议与规则结果比对，计算 TrustLevel，标记冲突。

#### 3a. 模板解析交叉验证（AI-1）

对每个 AI 返回的变量执行：

| 条件 | TrustLevel | 行为 |
|------|-----------|------|
| key 命中注册表 + 分类一致 | **HIGH** | 自动通过 |
| key 未注册，但别名匹配到标准 key | **MEDIUM** | 添加 `suggested_key`，提示"建议使用标准 key" |
| 已注册但 AI 分类与注册表不一致 | **MEDIUM** | 添加 warning "AI 分类与注册表不一致" |
| 完全未匹配（未注册 + 别名也无） | **LOW** | **强制审核**，不可跳过创建 |

在 TemplateStep 中有**硬性阻断**（`template_step.tsx:163`）：
```typescript
if (validVars.some((item) => item.trust_level === "low")) {
  toast.error("存在需审核的变量，请修正 key、采纳建议或删除后再创建")
  return
}
```

#### 3b. 去重交叉验证（AI-2）

将 AI 的去重建议与注册表别名关系比对：

| 条件 | TrustLevel |
|------|-----------|
| AI 建议的 merge 对与注册表别名**完全匹配** + confidence >= 0.8 | **HIGH** |
| AI 建议的 merge 对与注册表别名**完全匹配** + 0.5 <= confidence < 0.8 | **MEDIUM** |
| AI 建议与注册表别名**冲突**（两个 key 在注册表中无别名交集） | **LOW** |
| 仅 AI 建议，无规则支持 | confidence >= 0.5 → **MEDIUM**，否则 **LOW** |

注册表派生的别名建议（`_build_alias_suggestions`）**默认为 HIGH**，因为来源是确定性规则。

#### 3c. 校验结果交叉验证（AI-3）

合并正则校验 + AI 语义校验结果：
- 同时在正则和 AI 中都发现的问题 → 标记 `cross_validated: true`
- 仅 AI 发现的问题 → 标记 `source: "ai_only"`，单独标记

---

### 第4层：人工审核

**目标**：用户对 AI 结果拥有最终决定权。

#### 4a. 模板解析结果审核

用户在 `ParsedVariablesEditor` 中逐条浏览每个变量：
- 编辑 key / label
- **采纳建议**（点击"采纳建议"按钮，key 替换为 suggested_key，trust_level 升为 HIGH）
- **删除**不正确的变量
- 用户手动编辑 key 或 label 后，LOW 自动升为 MEDIUM（表示已人工处理）
- **LOW 变量必须审核通过才能创建模板**

#### 4b. 去重建议审核

用户在 `DedupSuggestionsPanel` 中逐条决定：
- **采纳**：系统自动合并变量值
- **拒绝**：该建议被丢弃

#### 4c. 变量值填写

Step 2 中用户手动填写每个变量的值，直接控制最终生成的内容。

#### 4d. 降级提示

当 AI 服务不可用时，系统**不崩溃**，展示 `AiDegradedBanner`，变量列表为空，用户可以**完全手工**添加和填写变量。

---

### 工作流全景（以模板上传为例）

```
用户上传 .docx
  ↓
[第1层] ai_service.parse_template_variables()
        → LLM 结构化输出（JSON Schema 约束）
        → 3次重试，失败则降级
  ↓
[第2层] template_service.enrich_parsed_variables()
        → 注册表匹配：已注册的覆盖 label/category/data_type
  ↓
[第3层] ai_guardrail.cross_validate_parsed_variables()
        → 别名匹配 → suggested_key
        → 分类一致性检查 → warnings
        → 计算 TrustLevel（HIGH/MEDIUM/LOW）
  ↓
[第4层] 前端 ParsedVariablesEditor 展示
        → 用户逐条查看、编辑、采纳建议、删除
        → LOW 变量不可跳过，必须处理
        → 点击"确认创建"前再次校验是否仍有 LOW
  ↓
        创建模板（所有变量通过审核）
```

---

### 补充数据说明

| 字段 | 所属层级 | 来源 | 含义 |
|------|---------|------|------|
| `trust_level` | 第3层 | 系统 Guardrail 计算 | 系统侧的可信评级 |
| `warnings` | 第3层 | 系统 Guardrail 计算 | 系统检测到的问题 |
| `suggested_key` | 第3层 | 别名匹配 | 建议使用的标准 key |
| `confidence` | 第1层 | LLM 自评 | LLM 对自己的提取有多自信 |
| `evidence_list` | 第1层 | LLM 自评 | 提取依据的原文片段 |
| `risk_note` | 第1层 | LLM 自评 | LLM 识别的风险点 |
| `is_registered` | 第2层 | 注册表查询 | 变量命中了注册表 |
| `cross_validated` | 第3层 | 正则+AI 交叉 | 校验问题被双方同时确认 |

## 五、生成体系

### 异步生成模式

```
用户点"生成" → 创建 GenerationTask → ThreadPoolExecutor / Temporal Worker 处理
                ↓
           前端指数退避轮询（2s → 4s → 8s → max 10s）
```

| 模式 | 启用条件 | 特点 |
|------|---------|------|
| ThreadPoolExecutor | `USE_TEMPORAL=false`（默认） | 进程内线程池，简单无依赖 |
| Temporal Workflow | `USE_TEMPORAL=true` | 声明式重试、Web UI 可见性、跨节点容错 |

### 双模式架构

系统核心链路（LLM 调用 + 文档生成）均支持双模式：

| 组件 | 直接模式 | Temporal 模式 |
|------|---------|--------------|
| AI 调用 | 手写 for 循环 3 次重试 | `AIActivityWorkflow` 声明式重试 |
| 文档生成 | `ThreadPoolExecutor` + `threading.Event` | `DocumentGenerationWorkflow` + Signal 取消 |

切换方式：设置环境变量 `USE_TEMPORAL=true`，Temporal 不可用时自动降级到直接模式。

### 取消生成逻辑

当用户点击"取消"：
1. 普通模式：设置 `threading.Event` → Worker 线程每次生成完一个文件检查 → 取消
2. Temporal 模式：发送 `cancel` Signal → Workflow 收到后终止剩余活动
3. 两种情况都会：清理已生成的部分文件 → 更新 DB 状态为 `cancelled` → 项目状态恢复为 `draft`

## 六、常见系统行为问答

**Q: 为什么变量不直接存 JSON 数组，而要拆成多行？**  
A: 方便逐行乐观锁更新和前端异步编辑。如果用 JSON 数组，两个用户同时增删经办律师会整段覆盖。

**Q: 预置模板为什么不能删除？**  
A: 它们是系统基础的"三件套"，删除后新项目再无标准模板可选。自定义模板满足个性需求，预置模板保障基础能力。

**Q: 模板刷新会不会把用户已经填好的变量值弄丢？**  
A: 不会。刷新只变更 Variable 的增删和 required 状态，**不影响已有变量的 value**。旧模板删掉的变量，如果其他模板还在引用，也不会被删除。

**Q: AI 降级是什么？**  
A: 当 AI 服务不可用（LLM 请求超时/重试耗尽），系统不会崩溃——解析返回空变量列表并提示"AI 服务不可用"，用户可以手动逐条添加变量。

**Q: 什么是"source_template_ids"？**  
A: 记录该变量是从哪些模板来的。移除模板时，该模板 ID 从列表中移除；列表为空时删除变量行。这是"为什么删除模板会自动清理不需要的变量"的实现基础。
